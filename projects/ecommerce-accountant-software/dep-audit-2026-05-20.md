## 3. Specific-points investigation

### 3.1 `xlsx` SheetJS — the "moved off npm" question

**Installed**: `xlsx@0.18.5`, resolved from `https://registry.npmjs.org/xlsx/-/xlsx-0.18.5.tgz`. License: Apache-2.0. **It is still on the registry** — the package was never *unpublished*, but **no patched version was ever published to npm**. SheetJS the company moved to https://cdn.sheetjs.com/ for new releases (0.20.x+), and `npm install xlsx` from the registry still pins to 0.18.5 with no patch path.

Both advisories (GHSA-4r6h-8v6p-xvw6 prototype pollution + GHSA-5pgg-2g8v-p4x9 ReDoS) apply to 0.18.5. The `npm audit` output reports `fixAvailable: false` for exactly this reason: the patched 0.19.3 / 0.20.2 / 0.20.3 builds exist only on the SheetJS CDN, not the public registry.

**Options to remediate**:

1. **Switch to `exceljs`** (MIT, actively maintained, on the public registry, multi-million weekly downloads). API is different, so this is a refactor of `importer.service.ts` (one file).
2. **Switch to `sheetjs-style` or a community fork** — riskier, lower-quality maintenance.
3. **Pull the patched SheetJS Community build from the CDN/git tag**: `npm install https://cdn.sheetjs.com/xlsx-0.20.3/xlsx-0.20.3.tgz`. Works but pins to a non-registry URL — bad for reproducibility, and the SheetJS docs say the public-CDN distribution is the supported Community channel.
4. **Accept risk** with a compensating control: input-validation layer around the `XLSX.read(buffer, …)` call (strict-types-only `cellDates: true, cellText: false, sheetRows: <cap>`), file-size cap, allowlist of expected sheet names. Existing auth gate (admin-only) is the main mitigation; documenting it as an accepted risk is acceptable if upgrade is genuinely too expensive.

Given the Pro-tier paywall on the official fix path and the active maintenance of `exceljs`, option (1) is the clean recommendation.

### 3.2 Next.js 16.2.4

13 advisories, all fixed in 16.2.5 or 16.2.6 — see § 2.1 for the full table. The blast radius is large because Next 16 dropped at the end of 2025 and a flurry of post-release fixes shipped through 16.2.6. Practical exposure on this single-tenant ERP depends on which features are used: SSRF (GHSA-c4j6-fc7j-m34r) requires WebSocket upgrade handling; the middleware-bypass set requires a middleware. **Two of the bypasses (GHSA-26hh-7cqf-hhc6 and GHSA-3g8h-86w9-wvmq) are only fully fixed in 16.2.6**, so the recommendation isn't 16.2.5 — go to 16.2.6.

### 5.4 Practical posture

- **No banned licenses.** No GPL-2.0 / GPL-3.0 standalone, no AGPL, no CDDL, no EPL, no proprietary anywhere.
- **MPL-2.0 (lightningcss, axe-core)** — file-level copyleft. Using the library as a dependency without modifying it does **not** trigger MPL's source-disclosure obligation. Standard practice in commercial codebases.
- **LGPL-3.0-or-later (sharp/libvips)** — dynamically-linked native binaries, used as opaque libraries. LGPL allows commercial use as long as the LGPL library itself remains replaceable and its license text is distributed alongside. For a server-side SaaS that ships no binaries to end users, this is generally compliance-clean.
- **`Apache-2.0 AND LGPL-3.0-or-later`** — compound expressions in the sharp/libvips wasm and Windows builds. Both terms must be honoured; the Apache-2.0 part adds attribution + patent-grant obligations.

**Recommendation**: this is a "send to Legal for a 30-minute review" file, not a "stop the build" finding. Single-tenant SaaS deployment + no end-user binary distribution + transitive-only usage = typical commercial profile. Get a one-line legal sign-off and move on.
